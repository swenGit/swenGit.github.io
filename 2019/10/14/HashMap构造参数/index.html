想写demo,先了解一下HashMap的扩容时机吧

扩容时机:

看下代码,以jdk1.8为例,默认大家都看过源码了,借此回顾一下

put(k,v)

大概总结下:

​ 前边的逻辑很简单,大概就是先定位桶的位置,然后为空则newNode(size相关), 不为空就找e–key相同的节点,找不到就newNode,找到了就覆盖value, 然后进行并发和扩容判断

newNode关注一下里边的size变化:

newNode()

看看size吧

size

也就是说put中没有其他size操作,只有 ++size了…

逻辑非常简单,如果size>768就扩容, 注意, 是size,意思是 The number of key-value mappings,每次put的时候都会自增的,而不是之前想的桶的数量

关于threshold:阈值

在resize中值为: float ft = (float)newCap * loadFactor; newCap = oldCap << 1; ft再转int,也就是常说的 1024*0.75

那桶的数量呢?

看一下吧,放进1024个mappings后的capacity是多少

Demo

看一下空桶的数量

不难嘛,记错了,之前看简介没看代码,以为是桶的数量要少于元素的数量,原来不是那么回事,实际上反而会多于元素数, 也就是说平时的table中会有大量空桶, 为什么要设置个过载因子呢? 或者为什么要用0.75,而不是1呢?

可以看到,空桶的数量是相当多的,如果改成HashMap<String, String> map = new HashMap<>(1024 ,1);后UUID结果总容量:1024 空桶数量:379 有效桶数量:645,hash冲突次数提高了379-205=174次,也就是put/get爬链表的机会变多了,效率有所下降,对应的是table长度减少.

据说0.75是数学计算出来的(类似31或者hash的>>>16取亦或),为了平衡时间和空间

HashMap的loadFactor为什么是0.75? https://www.jianshu.com/p/64f6de3ffcc1

继续玩, 设置HashMap<String, String> map = new HashMap<>(512,2F);会让hash冲突更激烈
